# <a name="analytics-platform-system-processing-and-storage-capacity"></a>分析平台系統的處理和儲存體容量
您的業務需求決定數目資料擴充單元，以及您需要在您 Analytics Platform System (APS) 的應用裝置中的計算節點磁碟的大小。 您可以使用這些處理和儲存體的計算，引導您購買與規劃決策的容量。  
  
  
## <a name="section1"></a>處理容量規劃  
查詢效能的 SQL Server Parallel Data Warehouse (PDW) 高度取決於您的資料，以平行方式處理的 CPU 核心數目。 限制範圍內，增加平行處理原則，將可改善大量平行處理 (MPP) 查詢效能。 即使您資料的大小是相對較小，MPP 查詢引擎的電源功能增強了具有更高的平行處理原則。  
  
例如，12 計算節點使用的應用裝置具有 192 處理您的資料，以平行方式的 CPU 核心。 這是 192 方法平行處理原則 ！ 使用 56 計算節點的應用裝置有 896 核心所有平行運作。 平行處理原則的這個大小不是可達成而不 MPP 計算中。  
  
計算節點的數目增加時，向外延展應用裝置需要新增多個計算節點，利用明顯的優點，一次。 硬體廠商支援只在特定的組態資料縮放單位，以確保調整設備的優點超過跨多個運算節點轉散發資料的成本。

說明：  
  
-   **磁碟機大小**1、 2 或 3 TB 為每個硬體廠商。  
  
-   **每個計算節點的實體資料存放區**= （磁碟機大小） * （16 個磁碟，每個計算節點）。 鏡像的磁碟未包含，因為它是以供備援。  
  
-   **最大的計算節點，每個機架**特有硬體廠商。  
  
-   **實體的最大資料儲存區，每個機架**= （每個計算節點的實體資料存放裝置） * （每個機架的最大的計算節點）。  
  
-   **估計每個機架的最大使用者資料儲存體**= （每個機架實體的最大資料儲存體） * (5:1 壓縮比率為 5) \* （50%的記錄檔和 tempDB)。 這是保守估計未壓縮的使用者資料可以載入並儲存至應用裝置。 這是估計值，而且不會強制執行軟體。 實際的使用者資料儲存體取決於您的資料和您的設定。  
  
-   **最大的機架**針對每個硬體廠商。  
  
-   **估計每個應用裝置的最大資料儲存體**= （每個機架估計最大資料儲存體） * （機架最大值）。 這是您無法載入，並且完整建置的應用裝置上儲存使用者資料的總計大小總計保守估計。

# <a name="what-is-a-cursor"></a>什麼是資料指標？
關聯式資料庫中的作業會針對完整的資料列集運作。 由 SELECT 陳述式所傳回的資料列集包括所有滿足陳述式 WHERE 子句之條件的資料列。 由陳述式傳回的完整資料列稱為結果集。 應用程式，尤其是互動式且在線上，不一定有效地運用整個結果集做為一個單位。 這些應用程式需要一個機制，一次運用一個資料列或小型資料列區塊。 資料指標就是一種結果集的擴充，提供此種機制。  
  
 資料指標是由資料指標程式庫實作。 資料指標程式庫是軟體，通常會實作成一部分的資料庫系統或資料存取 API，用來管理從資料來源 （結果集） 所傳回的資料屬性。 這些屬性包含並行管理，將結果集中傳回的資料列數目，以及是否可以向前或向後移動 （或兩者） 透過結果集 （以可捲動性）。  
  
 資料指標會追蹤結果集中的位置，並可讓您執行多個作業列針對結果集，不論傳回至原始的資料表。 換句話說，資料指標在概念上傳回資料庫中的資料表為基礎的結果集。 資料指標是如此命名因為它會指出在結果集中，目前的位置，就如同在電腦螢幕上的資料指標會指出目前的位置。  
  
 請務必熟悉的資料指標概念再繼續進行了解在 ADO 中其使用方式的特性。  
  
 使用資料指標，您可以：  
  
-   指定定位於結果集中的特定資料列。  
  
-   擷取一個資料列或根據目前的結果集位置的資料列的區塊。  
  
-   修改結果集目前位置的資料列中的資料。  
  
-   其他使用者所進行的資料變更定義不同層級的敏感度。  
  
 例如，請考慮會顯示清單中的可用產品潛在購買者的應用程式。 買方捲動清單，查看產品詳細資料與成本，並最後選取 購買的產品。 其他的捲動和選取範圍就會發生清單的其餘部分。 買方則擔心，只要產品會出現一次，但應用程式會使用可捲動資料指標向上和向下瀏覽的結果集。  
  
 您可以使用各種不同的方式中的資料指標：  
  
-   以完全沒有資料列。  
  
-   進行部分或全部的單一資料表中的資料列。  
  
-   進行部分或全部以邏輯方式聯結的資料表中的資料列。  
  
-   為唯讀或可更新資料指標或欄位層級。  
  
-   為順向的或完全可捲動。  
  
-   使用資料指標索引鍵集位於伺服器上。  
  
-   其他應用程式 （例如成員資格、 排序、 插入、 更新和刪除） 所造成的基礎資料表變更影響。  
  
-   現有的伺服器或用戶端上。  
  
 唯讀資料指標幫助使用者瀏覽的結果集，並讀取/寫入資料指標可以實作個別的資料列更新。 可以使用點回到基底資料表的資料列的索引鍵集，定義複雜的資料指標。 雖然某些資料指標是唯讀的順向方向，但其他人可以來回移動，以及提供動態的重新整理，根據其他應用程式會對資料庫進行的變更之結果集。  
  
 並非所有應用程式必須使用資料指標來存取或更新的資料。 只是某些查詢不需要使用資料指標來更新資料列導向。 資料指標必須是一項您選擇要擷取資料的最後一個技巧，然後您應該選擇最低的影響資料指標可能。 當您建立結果集所使用的預存程序時，結果集不是可更新資料指標使用編輯與更新方法。  
  
## <a name="concurrency"></a>並行  
 在某些多使用者的應用程式是要保持在最終端使用者顯示的資料非常重要。 這類系統的典型的例子為航空保留系統，許多使用者可能會爭奪給定的班機 （因此，單一資料錄） 上相同的基座。 在這種情況下，應用程式的設計必須處理並行作業的單一記錄。  
  
 其他應用程式中，並行不是那麼重要。 在這種情況下，無法對齊參與目前的資料隨時的保持費用。  
  
## <a name="position"></a>位置  
 資料指標也會追蹤的結果集內的目前位置。 將資料指標位置為目前的記錄，其方式類似於指標陣列索引會指向該陣列中的特定位置的值。  
  
## <a name="scrollability"></a>可捲動性  
 您的應用程式所採用的資料指標的類型也會影響的功能向前及向後移動的資料列結果集;這有時候稱為可捲動性。 向前移動的能力*和*回溯透過結果集增加的資料指標，複雜性，因此實作成本更高。 基於這個理由，您應該使用這項功能只在必要時要求資料指標。

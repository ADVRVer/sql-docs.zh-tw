# <a name="why-was-odbc-created"></a>為什麼建立 ODBC？
從歷史上看,公司使用單個 DBMS。 所有資料庫訪問都通過該系統的前端或通過編寫的應用程式來完成,這些應用程式完全用於該系統。 然而,隨著計算機的使用越來越多,計算機硬體和軟體越來越多,公司開始購買不同的DBMS。 原因很多:人們買了最便宜的,最快的,他們已經知道的,市場上最新的,什麼最適合一個單一的應用程式。 其他原因包括重組和合併,以前只有一個DBMS的部門現在有幾個。  
  
 隨著個人電腦的出現,這個問題變得更加複雜。 這些計算機帶來了一系列用於查詢、分析和顯示數據的工具,以及許多價格低廉、易於使用的資料庫。 從那時起,單個公司的數據通常分散在無數的桌面、伺服器和微型計算機中,存儲在各種不相容的資料庫中,並且由大量不同的工具訪問,其中很少可以獲取所有數據。  
  
 最後的挑戰來自用戶端/伺服器計算的出現,它尋求最有效地利用計算機資源。 廉價的個人計算機(用戶端)位於桌面上,同時提供數據的圖形前端和一些廉價的工具,如電子表格、圖表程式和報表生成器。 微型電腦和大型計算機(伺服器)承載 DBMS,在那裡他們可以使用其計算能力和中心位置提供快速、協調的數據訪問。 那麼前端軟體是如何連接到後端資料庫的呢?  
  
 獨立軟體供應商 (ISV) 也面臨著類似的問題。 為小型電腦和大型機編寫資料庫軟體的供應商通常被迫為每個 DBMS 編寫一個版本的應用程式,或為他們想要訪問的每個 DBMS 編寫特定於 DBMS 的代碼。 為個人計算機編寫軟體的供應商必須為他們想要使用的每個不同的 DBMS 編寫數據訪問例程。 這通常意味著大量的資源用於編寫和維護數據訪問例程而不是應用程式,而應用程式通常不是以品質出售,而是銷售它們是否可以訪問給定 DBMS 中的數據。  
  
 這兩組開發人員都需要一種訪問不同 DBMS 中數據的方法。 大型機和微型計算機組需要一種方法來合併單個應用程式中來自不同 DBMS 的數據,而個人計算機組需要此功能以及編寫獨立於任何一個 DBMS 的單個應用程式的方法。 簡而言之,兩個組都需要一種可互操作的方式來訪問數據;他們需要開放的資料庫連接。
